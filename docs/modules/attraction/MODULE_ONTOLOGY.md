## Что Такое Онтология

Онтология модуля описывает не экран, не CRM-схему и не набор отчетов, а рабочую
реальность компании:

- какие объекты существуют;
- какие роли и ответственности действуют вокруг этих объектов;
- в каких состояниях объекты могут находиться;
- какие переходы между состояниями допустимы;
- какие факты доказывают, что переход действительно произошел;
- кто или что имеет право считать состояние достигнутым;
- где норма, а где отклонение, дефект данных или дефект процесса.

`Schema` говорит, какие поля может хранить база. `Ontology` говорит, что эти поля
означают в бизнес-реальности и какие действия из них следуют.

Для агентной разработки это governance-layer: агент должен понимать company
model до того, как проектировать отчеты, workflow, evals, интерфейсы или
автоматизацию.

## Слои Онтологии

| Слой | Вопрос | Что фиксируется в модуле |
| --- | --- | --- |
| Descriptive layer | Что существует? | Потенциальный участник, сделка привлечения, встреча, конверсионное событие, договорные условия, передача, отказ, возврат. |
| Dynamic layer | Что меняется во времени? | Состояния участника, стадии сделки, статусы посещений, переходы, сроки, протухание эффекта события, потери. |
| Kinetic layer | Как принимаются решения? | Правила перевода между состояниями, доказательства перехода, исключения, ownership решения, допустимые shortcut-маршруты. |

Если kinetic layer не описан, система видит данные, но не понимает реальную
логику действия. Это главный риск для будущих агентов.

## Главная Рамка

| Параметр | Определение |
| --- | --- |
| Модуль | `Привлечение` |
| Главный объект | `Потенциальный участник` |
| Процессная оболочка | `Сделка в воронке "Привлечение"` |
| Цель процесса | Провести потенциального участника от входящей базы до передачи в клуб через ценностный движок. |
| Основной механизм нового привлечения | Конверсионный движок событий: события меняют состояние участника и двигают его вправо по карте состояний. |
| Успешный финал | `Передано в клуб` |
| Текущие альтернативные исходы сделки | `Корзина`, `Прогрев`, `Возврат в Лидген(неквал)`, `Отклонено потребителем` |
| Не часть бизнес-онтологии | UI, комментарии дашборда, Paperclip, GitHub, продуктовые задачи, `lead_id` как технический CRM-идентификатор. |

Если первая сделка ушла в проигрыш, это не означает, что потенциальный участник
навсегда потерян для клуба. `Корзина`, `Прогрев` и `Возврат` фиксируют текущий
исход конкретной сделки или поставки. Целевая траектория при повторной работе
остается прежней: довести человека до передачи в клуб, если ценность и
релевантность подтверждены.

## Граница Модуля

Внутри онтологии:

- потенциальные участники, попавшие в контур привлечения;
- сделки Bitrix24 в воронке `Привлечение`;
- состояния воронки и решения переходов;
- встречи как отдельные повторяемые объекты;
- конверсионные события и их эффект на участника;
- договорные условия, график оплат и коммерческое оформление;
- передача в клуб;
- отказы, потери и возврат в лидген;
- агрегированная аналитика по безопасным локальным данным.

Снаружи онтологии:

- лидген как отдельный производственный контур до передачи в привлечение;
- партнерские лидогенерационные мероприятия как источник холодной аудитории;
- CTU и программы как возможные форматы, если они допроектированы под нужный
  конверсионный эффект;
- клубный/активационный контур после успешной передачи;
- продуктовые задачи, интерфейсные комментарии, Paperclip/GitHub workflow;
- персональные данные: имена, телефоны, email, raw comments, raw payloads,
  raw files.

## Карта Контуров Компании

| Контур | Что делает | Связь с привлечением |
| --- | --- | --- |
| Лидген | Добывает и первично квалифицирует аудиторию. | Передает подходящих потенциальных участников в `База входящая` или возвращает неквалифицированных. |
| Партнерские лидген-мероприятия | Дают холодную аудиторию и спрос. | Не являются производственной системой привлечения; после них нужен перевод в звонок-знакомство и конверсионный маршрут. |
| Привлечение | Ведет потенциального участника к клубу через знакомство, события, договор и handoff. | Основной модуль этой онтологии. |
| Конверсионный движок | Создает управляемые события и форматы, которые меняют состояние участника. | Инструмент движения участника вправо по карте `Активация -> Демонстрация -> Проблематизация -> Адмиссия`. |
| CTU / программы | Могут быть продуктами или форматами. | Становятся частью привлечения только когда допроектированы под целевой переход и доказуемый конверсионный эффект. |
| Клуб / сообщество | Принимает участника после успешного привлечения. | Получает участника через объект `Передача в клуб`. |
| Оплата / договор | Оформляет коммерческую часть вступления. | Поддерживает переход `Адмиссия -> Контракт -> На передаче`. |

## Основные Объекты

| Объект | Бизнес-смысл | Ключевые данные |
| --- | --- | --- |
| `Потенциальный участник` | Кандидат, которого рассматривают для клуба. Аналитический субъект модуля. | Источник, качество, клуб/филиал, ответственные, встречи, события, договор, исход. |
| `Сделка в воронке "Привлечение"` | Рабочий процесс вокруг одного потенциального участника. CRM-контейнер, но не главный бизнес-субъект. | Текущая стадия, история стадий, поля карточки, договор, отказ, передача. |
| `Звонок-знакомство` | Первый квалификационный контакт, где менеджер понимает готовность: закрывать сразу или вести через события. | Переход на стадию, дата первого касания, звонок/задача/контакт. |
| `Встреча` | Отдельный повторяемый объект, не равный стадии. | Встреча 1/2/3, дата, тип, место, календарь, куратор, статус, результат. |
| `Конверсионное событие` | Управляемое воздействие, которое должно изменить состояние участника и двинуть его вправо. | Целевое состояние, формат, дата, тип события, участие, результат, следующий шаг. |
| `Формат` | Конкретный способ события: гостевая, нетворкинг, программа, разбор, встреча. | Механика влияния, целевая аудитория, конверсионный эффект, срок действия эффекта. |
| `Договорные условия` | Коммерческая часть вступления. | Тариф, тип договора, вид платежа, продавец, дата договора, дата оплаты. |
| `График оплаты` | Рассрочка или платежный план. | Суммы и даты платежей 1-4. |
| `Передача в клуб` | Финальный handoff в клубный контур. | Критерий принятия, комментарий для сообщества, подтверждающие материалы. |
| `Отказ / потеря` | Текущий проигрыш конкретной сделки. | Причина отказа, детализация, стадия потери, следующий возможный маршрут. |
| `Возврат в Лидген(неквал)` | Возврат поставщику/Лидген УС, не обычная корзина. | Причина возврата, контекст возврата, маршрут. |

`Потенциальный участник` и `Сделка в воронке "Привлечение"` нельзя
склеивать. Один описывает человека как бизнес-объект процесса, второй описывает
CRM-оболочку и источник фактов.

## Роли

| Роль | Смысл |
| --- | --- |
| Потенциальный участник | Кандидат на вступление в клуб. |
| Менеджер привлечения | Ведет участника по процессу, принимает или инициирует переходы. |
| Лид-менеджер | Предыдущий или исходный владелец/источник участника в карточке. |
| Куратор встречи | Отвечает за организацию и качество встречи. |
| Клуб / принимающая команда | Принимает участника после handoff и подтверждает соответствие. |
| Владелец формата / программы | Проектирует или поставляет формат, который может давать конверсионный эффект. |
| Владелец онтологии | Удерживает определения объектов, состояний, переходов и правил решений. |

Application roles вроде `leader`, `employee`, `super_admin` относятся к доступам
продукта. Это не бизнес-роли привлечения.

## Состояния Конверсионного Движка

Это не просто типы мероприятий и не просто CRM-стадии. Состояние
конверсионного движка фиксирует достигнутый эффект у потенциального участника:
что изменилось в его восприятии, мотивации и готовности.

| Порядок | Состояние | Бизнес-смысл | Что должно измениться у участника |
| ---: | --- | --- | --- |
| 1 | `Активация` | Участник впервые погружается в среду и получает эмоциональный/социальный импульс. | «Мне интересно, здесь есть энергия, хочу смотреть дальше». |
| 2 | `Демонстрация` | Участник понимает, как работает клуб и что именно происходит внутри. | «Я вижу, как это устроено и какие форматы мне могут быть полезны». |
| 3 | `Проблематизация` | Участник осознает свою проблему и цену бездействия. | «У меня есть задача/боль, и клуб может помочь ее закрыть». |
| 4 | `Адмиссия` | Возникает взаимное ощущение соответствия. | «Здесь свои, я подхожу им, они подходят мне». |

### Активация

| Параметр | Описание |
| --- | --- |
| Цель | Перевести участника из холодного/нейтрального состояния в заинтересованное. |
| Механика влияния | Погружение в среду, вау-эффект, вдохновение, социальное доказательство. |
| Типовой результат | Участник хочет продолжить и готов на следующий шаг. |
| Примеры форматов | Гостевая встреча, нетворкинг-сессия, программа построения сообществ как возможный активационный формат. |
| Доказательства | Пришел на событие, положительно отреагировал, согласился на следующий контакт, проявил интерес к продолжению. |
| Риск | Эффект быстро протухает, если не сделать следующий шаг. |
| Следующий переход | `Активация -> Демонстрация` или `Активация -> Проблематизация`. |

### Демонстрация

| Параметр | Описание |
| --- | --- |
| Цель | Показать, как клуб работает на практике. |
| Механика влияния | Участник видит форматы, среду, людей и механику взаимодействия. |
| Типовой результат | Участник понимает устройство клуба и может примерить его на себя. |
| Главный вопрос | «Можно увидеть глазами, как это работает?» |
| Доказательства | Участник понимает форматы, задает прикладные вопросы, выбирает интересный маршрут, готов обсуждать пользу. |
| Риск | Участник увидел красивое мероприятие, но не связал это со своей задачей. |
| Следующий переход | `Демонстрация -> Проблематизация` или `Демонстрация -> Адмиссия`. |

### Проблематизация

| Параметр | Описание |
| --- | --- |
| Цель | Перевести интерес в осознанную потребность. |
| Механика влияния | Помочь участнику увидеть проблему, последствия бездействия и связь с возможностями клуба. |
| Типовой результат | Участник понимает, зачем ему клуб, а не просто интересуется. |
| Важное замечание | Это самое сложное состояние, но если оно достигнуто, участник близок к оплате. |
| Возможная сильная тема | Социальный капитал как рамка для боли и ценности. |
| Доказательства | Участник формулирует проблему, признает цену бездействия, спрашивает как клуб ее закрывает, готов обсуждать вступление. |
| Риск | Продавать продукт вместо интерпретации боли и ценности. |
| Следующий переход | `Проблематизация -> Адмиссия` или `Проблематизация -> Контракт`. |

### Адмиссия

| Параметр | Описание |
| --- | --- |
| Цель | Проверить взаимное соответствие участника и клуба. |
| Механика влияния | Ценностная сверка, ощущение «свои», подтверждение релевантности с обеих сторон. |
| Типовой результат | Участник чувствует, что это его среда; команда чувствует, что он подходит. |
| Важное уточнение | Адмиссия не сводится к деньгам или формальному отбору. |
| Доказательства | Участник принят как релевантный, готов к оформлению, команда подтверждает fit. |
| Риск | Считать адмиссию только проверкой платежеспособности. |
| Следующий переход | `Адмиссия -> Контракт -> На передаче`. |

## Конверсионное Событие

Конверсионное событие - это управляемое воздействие на потенциального участника,
которое должно изменить его состояние в конверсионном движке и перевести его
вправо по карте:

```text
Активация -> Демонстрация -> Проблематизация -> Адмиссия -> Контракт -> Передача
```

| Атрибут события | Что фиксирует |
| --- | --- |
| Целевое состояние | На какое состояние событие должно повлиять. |
| Целевой переход | Какой переход оно должно вызвать. |
| Формат | Конкретный формат: встреча, программа, гостевая, нетворкинг, разбор. |
| Температура аудитории | Холодная, теплая, уже готовая. |
| Механика влияния | Через что событие меняет состояние участника. |
| Доказательство эффекта | Какие факты показывают, что состояние достигнуто. |
| Срок действия эффекта | Когда эффект считается протухшим. |
| Следующий шаг | Куда участника вести сразу после события. |

| Класс события | Смысл | Связанное состояние |
| --- | --- | --- |
| `Активационное событие` | Первый значимый опыт клуба. | `Активация` |
| `Демонстрационное событие` | Показ ценности клуба/формата. | `Демонстрация` |
| `Проблематизационное событие` | Выявление или усиление запроса. | `Проблематизация` |
| `Адмиссионное событие` | Проверка соответствия и готовности. | `Адмиссия` |

### Статусы Участия В Событии

| Состояние участия | Смысл | Текущее техническое соответствие |
| --- | --- | --- |
| `Приглашен` | Участнику предложили событие. | Visit exists or entered `Приглашен` / `invited`. |
| `Подтвердил` | Участник выразил намерение участвовать. | Visit history contains `Пойду` / `confirmed`. |
| `Пришел` | Есть факт участия. | Visit history contains `На мероприятии` / `attended`. |
| `Отказался` | Участник отказался. | Visit history contains `Отказ` / `refused`. |
| `Не пришел` | Приглашение/намерение не стало участием после даты события. | Derived after event when no `На мероприятии` and final state is non-attended. |
| `Неизвестно` | Данных недостаточно. | Unknown/unmapped stage. |

Если посещение сначала было в `Отказ`, но позже перешло в `На мероприятии`, оно
считается `Пришел`, а не `Не пришел`.

## Правила Конверсионных Событий

| Правило | Смысл |
| --- | --- |
| Формат не равен состоянию | Один формат может активировать, демонстрировать и проблематизировать в разных пропорциях. |
| Событие должно двигать вправо | Если мероприятие не переводит участника в следующее состояние, оно не конверсионное для привлечения. |
| Эффект состояния временный | Активация, демонстрация и проблематизация со временем протухают. |
| Готовых не гоняем по цепочке | Если участник уже готов вступать, его нужно закрывать, а не искусственно вести через все события. |
| Партнерский лидген не равен привлечению | Партнерские мероприятия дают холодную аудиторию, которую потом нужно переводить в звонок-знакомство и конверсионный маршрут. |
| CTU/программы могут стать форматами | Только если их допроектировать под нужный переход и конверсионный эффект. |
| Событие без следующего шага дефектно | Конверсионное событие должно иметь route after event: встреча, звонок, адмиссия, контракт или возврат. |

## Встреча Как Объект

Встреча - отдельный объект, не стадия и не просто поле.

| Слот | Смысл |
| --- | --- |
| `Встреча 1` | Первая существенная встреча. |
| `Встреча 2` | Повторная встреча для продолжения движения. |
| `Встреча 3` | Дополнительная встреча перед решением. |

| Атрибут встречи | Что фиксирует |
| --- | --- |
| Дата и время | Когда встреча назначена или прошла. |
| Тип | Очная, Zoom, мероприятие. |
| Место | Офис, клиент, Zoom, внешняя площадка, мероприятие. |
| Календарь | Москва, СПБ, ПВБ или иной операционный календарь. |
| Куратор | Ответственный за встречу. |
| Статус | Запланирована, проведена, перенесена, отменена, не состоялась. |
| Результат | Следующий шаг после встречи. |

Правила:

- `Встреча-знакомство` требует объекта встречи или явных данных встречи.
- Встречи могут проходить не только на `Встреча-знакомство`.
- Встреча может относиться к активации, демонстрации, проблематизации или
  адмиссии.
- Стадия без обязательных данных встречи означает проблему данных или процесса.

## Договор, Оплата И Передача

| Объект | Смысл |
| --- | --- |
| `Договорные условия` | Тариф, тип договора, продавец, плательщик, договорные даты и файлы. |
| `Разбивка сумм` | Лицензия, адмиссия/регсбор, НДС-регсбор. |
| `График оплаты` | Суммы и даты платежей 1-4. |
| `Передача в клуб` | Критерий принятия, комментарий для сообщества, подтверждающие материалы. |

Эти объекты нельзя схлопывать в одно revenue-поле. Они отвечают на разные
вопросы: пакет, юридическая форма, продавец, плательщик, структура платежа,
cash-flow и готовность к handoff.

## Отказы И Альтернативные Маршруты

| Объект | Смысл |
| --- | --- |
| `Корзина` | Текущий проигрыш после работы КИ: клиенту сейчас не ценно, он отказался или не дошел до членства. |
| `Прогрев` | Отложенный повторный маршрут, когда человек сейчас не вступает, но есть смысл вернуться позже. |
| `Возврат в Лидген(неквал)` | Возврат поставщику/Лидген УС по неквалу или нарушению условий качества до принятой покупки. |
| `Отклонено потребителем` | Принимающий клуб не принял участника/карточку на этапе `На передаче`; это не отказ клиента. |
| `Причина отказа` | Структурная причина провала. |
| `Детализация отказа` | Дополнительный контекст, пригодный только в sanitized aggregate/review context. |
| `Альтернативное предложение` | Иной клуб, продукт или маршрут. |

`Корзина`, `Прогрев`, `Отклонено потребителем` и `Возврат в Лидген(неквал)`
нельзя объединять в один loss bucket: это разные решения, владельцы и следующие
действия.

## Решения Переходов

Это kinetic layer: не только какие состояния существуют, но и когда компания
имеет право считать переход состоявшимся.

| Переход | Решение | Основание |
| --- | --- | --- |
| `База входящая -> Звонок-знакомство` | Взять в квалификационную коммуникацию. | Стадия, блок `Взят в работу`, дата первого касания, звонок/задача. |
| `Звонок-знакомство -> Встреча-знакомство` | Вывести на обязательную встречу. | Встреча 1 или подтвержденные данные встречи. |
| `Встреча-знакомство -> Активация` | Подтвердить вовлечение после знакомства. | Проведенная встреча, активационное событие, решение менеджера. |
| `Активация -> Демонстрация` | Показать ценность. | Демонстрационное событие или встреча. |
| `Демонстрация -> Проблематизация` | Перейти к проработке запроса. | Встреча, событие, решение менеджера. |
| `Проблематизация -> Адмиссия` | Допустить к оценке соответствия. | Решение о готовности/релевантности. |
| `Адмиссия -> Контракт` | Перейти к оформлению. | Положительное решение, договорные условия. |
| `Контракт -> На передаче` | Подготовить handoff. | Договор/условия, критерии принятия. |
| `На передаче -> Передано в клуб` | Завершить привлечение. | Финальная передача. |
| `На передаче -> Отклонено потребителем` | Зафиксировать, что клуб не принял участника/карточку. | Решение принимающего клуба, полнота данных, критерии принятия. |
| Любое активное -> `Корзина` | Зафиксировать текущий проигрыш: клиенту сейчас не ценно или он отказался. | Причина отказа. |
| Любое активное -> `Прогрев` | Отложить повторную работу, если человек потенциально релевантен, но не готов сейчас. | Причина, срок возврата, следующая задача. |
| Любое активное -> `Возврат в Лидген(неквал)` | Вернуть поставщику/Лидген УС. | Причина возврата/маршрутизация. |

Допустимые shortcut-решения:

- если участник уже готов вступать, его можно вести сразу к `Контракт`, не
  прогоняя через все конверсионные события;
- если участник не подходит под критерии актуальных клубов/заказчиков, его можно
  вернуть в лидген или направить в отдельный контур вроде CTU, если такой маршрут
  явно определен;
- если состояние было достигнуто, но эффект протух, нужен повторный формат или
  новая активация, а не автоматическое продолжение воронки.

## Правила Аналитики

| Вопрос | Правильная база анализа |
| --- | --- |
| Обрабатывается ли база? | `База входящая -> Звонок-знакомство` плюс work-start/first-touch facts. |
| Доходит ли база до встречи? | Участники из базы с валидной встречей или переходом в `Встреча-знакомство` с данными встречи. |
| Надежна ли стадия `Встреча-знакомство`? | Стадия есть и встреча заполнена. |
| Двигают ли события участников? | Класс события + статус участия + следующий переход. |
| Где узкое место? | Время в состоянии, доля выхода дальше, доля потерь. |
| Какие источники качественные? | Источник -> встреча -> активация -> контракт -> передача. |
| Кто эффективно ведет процесс? | Скорость и качество переходов менеджера, не только финальный успех. |
| Почему участники выпадают? | Состояние потери + причина отказа + стадия на момент потери. |
| Какие форматы работают? | Формат -> целевое состояние -> доказательство эффекта -> следующий переход. |

## Data Quality Rules

- Использовать `deal_id` и безопасные внутренние IDs для агрегации.
- Не выводить deal names, contact names, phones, emails, raw payloads.
- Дашборды должны читать cached local API/SQLite data, а не Bitrix напрямую.
- Отчеты должны оставаться в attraction manager whitelist, если issue явно не
  меняет scope.
- Стадия без обязательных подтверждающих данных - сигнал качества данных, а не
  бизнес-доказательство.
- Низкоуверенные связи через contact/lead/event fallback нельзя использовать для
  жестких управленческих выводов без маркировки confidence.
- Исторические/legacy стадии должны оставаться legacy evidence, а не чистыми
  бизнес-состояниями.

| `Потенциальный участник` | лид, клиент, контакт |

| `Встреча-знакомство` | просто встреча |

| `Формат` | состояние участника |
