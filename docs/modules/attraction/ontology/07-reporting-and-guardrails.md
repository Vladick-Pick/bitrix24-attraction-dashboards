# 07. Отчетность И Ограничения Для Агентов

## Что Должны Видеть Отчеты

| Вопрос | Правильная база анализа |
| --- | --- |
| Быстро ли берут поставки в работу? | Поставки `Готов ко встрече`, переход `База входящая -> Звонок-знакомство`, SLA-1 и автопокупки. |
| Делается ли первый звонок? | Первый звонок или первая попытка связи в течение 1 рабочего дня, отдельно от stage move. |
| Доводит ли КИ до встречи? | Переход `Звонок-знакомство -> Встреча-знакомство` плюс данные встречи/задачи. |
| Сколько звонков относится к менеджеру Привлечения? | По умолчанию прямые звонки сделки плюс безопасный fallback через контакт с единственной сделкой; для менеджеров с политикой `direct_only` только прямая привязка звонка к сделке Привлечения. |
| Не покупаются ли лиды автоматически? | Доля `База входящая -> Встреча-знакомство` без ручного взятия в работу. |
| Какие источники качественные? | Источник -> итоговое качество -> встреча -> событие -> контракт -> передача. |
| Какие форматы двигают участников? | Формат -> целевой эффект -> участие -> следующий шаг -> переход дальше. |
| Сколько людей позвали на мероприятия и с каких этапов? | Отчет активности -> блок `Конверсионные мероприятия`: `event_visit_facts` + `deal_touchpoint_facts.kind = conversion_event_visit` + stage map онтологии. |
| Почему сделки проигрываются? | Маршрут проигрыша, причина из Битрикс, подробность, стадия на момент проигрыша. |
| Что уходит в прогрев? | Причины и сроки возврата по людям, которые сейчас не вступают, но не должны быть финально потеряны. |
| Что отклоняет клуб? | `Отклонено потребителем` после `На передаче`: полнота данных, релевантность, причина отказа принимающего клуба. |

## Метрики Стартового Регламента

| Метрика | Смысл |
| --- | --- |
| SLA-1: принятие в работу | Доля поставок, переведенных из `База входящая` вручную в течение 24 рабочих часов. |
| Автопокупки | Количество и доля сделок, автоматически переведенных на `Встреча-знакомство` из-за нарушения SLA-1. |
| SLA-2: первый контакт | Доля сделок, где первая попытка связи сделана в течение 1 рабочего дня. |
| SLA-3: цикл обработки | Доля сделок, где за 3 рабочих дня выполнены 2 звонка + 1 сообщение или оформлен корректный результат. |
| Возвраты | Количество и доля `Возврат (неквал)` по причинам. |
| Корзина | Количество и доля проигрышей после работы КИ. |
| Прогрев | Количество людей, отложенных для повторной работы, и наличие следующей задачи. |
| Отклонено потребителем | Количество карточек, не принятых клубом на этапе `На передаче`. |

## Data Guardrails

- Использовать `deal_id`, внутренние идентификаторы, stage ids и агрегаты.
- Не выводить имена, телефоны, email, raw payloads и сырые комментарии.
- Дашборды читают локальный API и SQLite snapshot, а не Bitrix напрямую.
- Bitrix sync - отдельная операция, не часть рендера отчетной страницы.
- Heavy reports загружать лениво или фоново, не блокируя стартовый UI.
- Отчеты должны оставаться в согласованном whitelist менеджеров привлечения,
  если отдельный issue явно не расширяет scope.

## Привязка К Отчету Активности

| Блок | Каноническая база | Что показывает |
| --- | --- | --- |
| `activities-calls` / `attraction-conversion-events` | Только прямые посещения мероприятий, связанные со сделкой Привлечения (`event_visit_deal`), без contact fallback. | `Мероприятие`, `Пригласили`, `Дошли`, `Отказ`, `Еще ждут`, `С каких этапов звали`. |

Для звонков отчет активности сохраняет две разные базы. `allCalls` показывает
полную нагрузку менеджера по синхронизированным звонкам, а `linkedDealCalls`
показывает звонки, которые можно отнести к сделкам Привлечения. Стандартная
политика менеджера допускает `contact_single_deal_fallback`, когда звонок
привязан к контакту с единственной сделкой Привлечения. Для менеджеров с
политикой `direct_only` такой fallback исключается из `linkedDealCalls`, потому
что часть их роли находится вне Привлечения; эти звонки остаются только в
`allCalls` и диагностике `excludedByPolicyCalls`.

Разрез `С каких этапов звали` использует этап сделки на момент приглашения
через canonical touchpoint. Scope менеджеров берется по владельцу сделки
Привлечения, а не по ответственному smart-process посещения: посещение может
быть заведено техническим пользователем Битрикс. Если один и тот же формат
зовет людей из `Звонок-знакомство`, `Активация`, `Демонстрация`, `Контракт` или
другого этапа, это не делает событие внешним к Привлечению; это показывает,
для какого перехода в конверсионном движке оно было использовано.

## Правила Для Агентов

| Правило | Значение |
| --- | --- |
| Не делать вывод только по названию стадии | Стадия отражает цель этапа; факт достижения требует подтверждений. |
| Не смешивать Лидген УС и Привлечение штрих | Это разные производственные контуры с контрактной точкой передачи. |
| Не считать партнерское мероприятие конверсионным автоматически | Нужен следующий шаг и эффект для потенциального участника. |
| Не подменять `Возврат (неквал)` корзиной | Это разные маршруты, разные владельцы смысла и разные причины. |
| Не считать `Отклонено потребителем` отказом клиента | Это отказ принимающего клуба на этапе `На передаче`. |
| Не считать `Прогрев` финальной потерей | Это отложенная повторная работа, если есть срок и контекст возврата. |
| Не использовать английские fact names в бизнес-доках | Бизнес-документы должны говорить нормальными русскими терминами. |
| При споре регламентов и Битрикс фиксировать расхождение | Битрикс актуальнее для values, регламент актуальнее для логики. |
