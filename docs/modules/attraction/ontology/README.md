# Онтология Привлечения

Этот каталог - рабочая версия онтологии модуля `Привлечение`.
Главный файл [MODULE_ONTOLOGY.md](../MODULE_ONTOLOGY.md) остается входной
страницей и большим снимком. Новые решения фиксируются здесь по разделам, чтобы
документ можно было обновлять точечно.

## Registry

Структурированный registry находится в
[registry/attraction-ontology.json](./registry/attraction-ontology.json). Его
читают валидатор, API и вкладка дашборда `Онтология`.

Иерархия источников:

1. Битрикс показывает фактическую настройку процесса.
2. Регламенты и таблицы являются источниками/evidence.
3. Онтология фиксирует каноническую трактовку.
4. При расхождении появляется drift, который классифицирует Технолог
   бизнес-процессов из Центра Технологизации.

## Правила Обновления

1. Не переписывать всю онтологию ради одного уточнения.
2. Обновлять самый узкий тематический файл.
3. Если меняется термин, добавить его в соответствующий раздел, registry и отметить в
   [журнале изменений](./CHANGELOG.md).
4. Если регламент и Битрикс расходятся, Битрикс считается источником актуальных
   значений полей и причин, регламент - источником evidence, а registry -
   канонической трактовкой для API и дашборда.
5. Не вводить технические fact names как бизнес-язык. В онтологии используются
   нормальные русские термины: `Лидген УС`, `взятие в работу`, `первый звонок`,
   `автопокупка`, `Возврат (неквал)`.
6. Обновлять registry при изменении стадий Bitrix, enum-причин, полей, SLA,
   логики отчетов или каталога конверсионных событий.
7. Перед PR запускать `pnpm ontology:validate`, а перед deploy просматривать drift
   во вкладке `Онтология`.
8. Контур Лидген УС не расширять в attraction registry; для него нужен отдельный
   будущий дизайн.
