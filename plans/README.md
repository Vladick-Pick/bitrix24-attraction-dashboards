## Dependency Notes

- 001 must land first because it records the decision every shared/platform
  follow-up must obey: `attraction` and `leadgen` are separate products, not two
  skins of one report.
- 002 should land before 003 because route registrars should depend on narrow
  storage roles, not on the full `SqliteRepository` adapter.
- 003 is the first code split that makes module disablement real at the HTTP
  interface without physically forking the repository yet.
- 004 should land after 002 and 003 so future modules and AI-agent consumers use
  explicit storage/route seams instead of direct repository or report internals.
- 005 should land after 004 because the manifest must exist before fork-owned
  modules can bring their own manifest adapter, live route availability, source
  data, and integration rules. It keeps the project source-code-fork friendly
  without introducing a package/plugin runtime yet.
- 006 is independent of the platform split plans, but depends on the active
  attraction manager catalog containing `7538` and `118`. It should land after
  the manager-addition branch or absorb that change explicitly.
- 007 is independent of the platform split plans. It should be implemented on a
  fresh `codex/unified-deal-timeline-card` branch and must keep the deal card
  scoped to local API/SQLite data, safe attraction fields and the existing
  dashboard design system. Execution completed on
  `codex/unified-deal-timeline-card`; blocks A-E are done, including focused
  suites, workspace checks, browser smoke and final CRG/code-review gate.
- 008 starts the multi-meeting-slot sequence. It must land before report or UI
  work because the current sync/storage shape can represent only one deal
  meeting field pair.
- 009 depends on 008 because meeting counts, lifecycle events, workload rows and
  economics must consume one canonical `meetingSlots` model and avoid double
  counting CRM meeting activities.
- 010 depends on 008 and 009 because the dashboard should render the final API
  shape instead of inventing client-side meeting counts.
- 011 depends on the module capability boundary from 004. It must complete the
  attraction report catalog before an agent gateway can safely execute reports.
- 012 depends on 011 because the gateway must use manifest report descriptors
  and `agentReadable` policy rather than maintaining a hidden allowlist.
- 013 depends on 011 and 012 because MCP should be a thin stdio protocol adapter
  over reviewed read-only gateway methods.
- 014 is independent of the platform and MCP plans. It must land before the
  calls/tasks/meetings heatmap branch is considered review-ready because the
  current expanded cards mix table-row metrics with different heatmap
  populations and off-grid counts.
- 015 starts the call enrichment feature sequence. It must land before code
  because the current repository security contract is read-only and explicitly
  forbids generic Bitrix writes.
- 016 depends on 015 because the allowlist and entity routing must implement the
  security decision, not invent a second contract in code. Later prompts,
  Telegram messages, diff logic, and Bitrix writeback must import this contract
  rather than duplicating field codes.
- 017 depends on 016 because proposal rows must store only approved field/entity
  references. It creates the lifecycle and audit substrate before intake,
  extraction, Telegram, or writeback.
- 018 depends on 017 so repeated Bitrix call events can be acknowledged
  idempotently and have a safe downstream state target.
- 019 depends on 018 because the cheap dialogue gate is used by the automatic
  intake path. It must run before full analysis to skip voicemail, robot,
  silence, and no-answer calls.
- 020 depends on 017, 018, and 019 because it owns the automatic state machine:
  intake -> dialogue gate -> full analysis -> enrichment hook. It must not send
  Telegram or write Bitrix directly.
- 021 depends on 016 and 020 because extraction must use the central field
  contract and run only after automatic analysis has produced transcript data.
- 022 depends on 016 and 021 because current-value reads and diff decisions need
  both the approved field descriptors and extraction candidates.
- 023 depends on 017 and 022 because Telegram should send only stored,
  meaningful proposals. It must not contain diff logic or Bitrix write logic.
- 024 depends on 015, 016, 022, and 023 because writeback is destructive and must
  apply only manager-approved, unexpired, allowlisted proposals after a fresh
  current-value check.
- 025 depends on the full feature chain because rollout modes and end-to-end
  tests prove dry-run, Telegram-only, limited-write, full V1, no-dialogue skip,
  no-op skip, deal/contact routing, and 48-hour expiry behavior.

## Findings Considered And Rejected

- Full repository fork now: rejected for this sequence because it would copy the
  current `createApp`, `SqliteRepository`, and `ProtoApp` coupling into two
  repositories.
- Full web runtime refactor now: deferred. `docs/architecture/web-runtime.md`
  requires `ProtoApp` to remain the one supported runtime interface; internal
  web adapters can come after server/storage seams.
- Full contracts package split now: deferred. It is useful, but less blocking
  than storage roles and route ownership for safe repository extraction.
- Package/plugin runtime now: deferred. Open-source forkability should first use
  a source-level extension seam; dynamic package loading can come later if real
  fork maintainers need it.
